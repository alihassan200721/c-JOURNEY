00
The Philosophy
Why C still matters after 30 years.

Every language I've worked in — from assembler at embedded shops, to C++ at game studios, to Rust at systems startups — everything reduces to C's mental model. Memory is a flat array. A pointer is just a number. The CPU doesn't know about objects.

My rule: never touch the keyboard until I can draw what the program does on paper. Flowchart first. Pseudocode second. Code third. This isn't slow — it's 10× faster because you debug on paper where it costs nothing, not in the terminal where it costs hours.

And you'll live in Neovim. Not because it's cool, but because when you control your editor at the muscle-memory level, your thinking and typing become one motion.

The Phases
Phase I
The Machine
2–3 Weeks
  • What a CPU actually does (fetch-decode-execute)
  • Memory: stack vs heap vs data vs code segments
  • Bits, bytes, binary, hex — no calculator
  • What a compiler actually produces
  • Your first gcc -S — read the assembly
Phase II
C Syntax & Control
3–4 Weeks
  • Types, variables, operators — all of them
  • if/else, switch — flowchart every branch
  • while, for, do-while — loop invariants
  • Functions: the stack frame model
  • Scope, lifetime, storage classes
Phase III
Pointers & Memory
4–6 Weeks
  • Pointer arithmetic — draw every step
  • Arrays as pointers — the decay rule
  • malloc / calloc / realloc / free
  • Double pointers, pointer-to-function
  • valgrind — zero leaks, zero UB
Phase IV
Structs & Data
3–4 Weeks
  • struct padding & alignment — know the bytes
  • Linked lists, stacks, queues from scratch
  • Binary trees, BST operations
  • File I/O: fopen, fread, fwrite, fclose
  • Bitfields and unions
Phase V
The Toolchain
2–3 Weeks
  • gcc / clang flags: -Wall -Wextra -fsanitize
  • Make & Makefiles — dependency graphs
  • gdb / codelldb: breakpoints, watch, backtrace
  • Header files, multiple .c files, linking
  • Static & shared libraries (.a / .so)
Phase VI
Real Projects
Ongoing
  • Shell in C (fork, exec, pipe, wait)
  • HTTP server from raw sockets
  • Memory allocator (implement malloc)
  • Read K&R — every exercise
  • Read a real codebase (Redis, SQLite)

03
Pseudocode Examples
EXAMPLE 1 — Pointer + Loop (the mental model)
// PSEUDOCODE — before writing a single line of C // Goal: sum all elements of an array using a pointer DEFINE array [5] = {10, 20, 30, 40, 50} DEFINE ptr → points to array[0] DEFINE sum = 0 DEFINE i = 0 WHILE i < 5: sum = sum + VALUE_AT(ptr) // dereference: *ptr ADVANCE ptr by 1 element // ptr++ moves sizeof(int) bytes i = i + 1 END WHILE PRINT sum /* Memory picture (draw this on paper first): addr: 0x100 0x104 0x108 0x10C 0x110 value: 10 20 30 40 50 ^ ptr starts here, walks right */
EXAMPLE 2 — Dynamic memory + struct
// PSEUDOCODE — linked list node insertion STRUCT Node: data : int next : pointer to Node FUNCTION insert_front(head, value): // Ask OS for memory for one Node new_node = ALLOCATE(sizeof Node) IF new_node == NULL: ABORT("out of memory") new_node.data = value new_node.next = head // new node points to old head head = new_node // head now points to new node RETURN head FUNCTION free_list(head): WHILE head != NULL: tmp = head.next FREE(head) // always free before losing pointer head = tmp END WHILE // RULE: every malloc must pair with exactly one free.
EXAMPLE 3 — The mental model for a C program's life
// What actually happens when you run ./program OS LOADS binary into memory: .text segment → your compiled instructions (read-only) .data segment → initialized globals .bss segment → zero-initialized globals stack → grows DOWN from high address (local vars, return addrs) heap → grows UP from low address (malloc land) CALL _start (C runtime, not main) → zero out .bss → set up argc, argv → CALL main() INSIDE main(): EACH function call: push return address onto stack push local variables onto stack execute body pop frame → locals are GONE EACH malloc(): kernel finds free heap page returns pointer (just a number) YOU are responsible for free() RETURN from main → OS reclaims everything // But forgetting free() is still a bug — train the habit

Start with clangd + treesitter only. Add plugins when you feel actual friction — not because a YouTube video told you to. The plugin that solves a real problem becomes muscle memory instantly. Plugins installed "because best practice" become noise.
05
Daily Ritual
Session Start · 15 min
Paper + Pen First
Write today's concept in one sentence. Draw the memory layout. Write pseudocode. Only then open Neovim.
Core Block · 60–90 min
Implement One Thing Completely
Not three half-finished things. One function, one data structure, one concept — compiled, tested, zero warnings, zero leaks.
Break · 10 min
Walk. No screens.
Your brain consolidates while you move. Non-negotiable. The solution to 80% of bugs comes on the walk.
Review · 15 min
Read Your Code Aloud
If you can't explain every line out loud — pointer arithmetic, the why of each cast — you don't understand it yet. Rewrite until you can.
End of Day · 10 min
Write One Insight
One thing learned, one mistake made, one question to answer tomorrow. A text file in your repo. Build a lab notebook habit early.

Core Mental Models
🧠
Memory Is A Tape
RAM is one giant byte array. Every variable, every pointer, every function lives at a numeric address. There is no magic — only addresses and values at those addresses.
📦
Types Are Lenses
The same 4 bytes can be an int, a float, or a pointer. The type tells the compiler how to interpret the bits. Cast recklessly, get chaos. Cast intentionally, get power.
📍
Pointers Are Addresses
A pointer is just a number — the address of something in memory. Pointer arithmetic is just integer arithmetic in units of sizeof(type). Draw it every time.
📚
Stack Frames Are LIFO
Every function call pushes a frame onto the stack. Local variables live there. When the function returns, the frame is gone. Never return a pointer to a local variable.
⚙️
Undefined Behavior Is Real
UB isn't "might work on this machine." It means the compiler can assume it never happens and optimize accordingly — deleting the very code you relied on. Use -fsanitize=address,undefined always.
🔗
Ownership Is Yours
C has no GC. Every malloc is a contract: you own this memory, you will free it, exactly once, after its last use. Track ownership explicitly — in comments if needed.
